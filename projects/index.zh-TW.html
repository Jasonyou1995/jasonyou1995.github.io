01 · Agent 平台

OC1: Agent 安全控制系統

多供應商 Agent 棧, 配一個用形式化方法驗證過的安全迴路。

狀態已交付 · 生產級
技術棧Python · FastAPI · TLA+ · Solidity
規模18,400+ 行 · 7 份合約 · 14 套測試
JD 對應Agent Harness · 評測 · Trace · 人在環

問題

前沿模型呼叫會以三種 Demo 看不到的方式失敗: 供應商宕機、提示注入、靜默策略漂移。聊天機器人 Demo 不需要處理這些, 但生產級 Agent 系統必須 fail safe, 不能只是 fail loud。

動機

我要的是一個敢把錢和真實用戶放上去的底盤。意味著要有一個 可證明 安全的 kill-circuit, 而不是一個希望它安全的迴路, 還要讓每一次外部呼叫都可觀測、可回滾。

具體造了什麼

跨 OpenAI、Anthropic、本地 Ollama 的多供應商 LLM 編排, 含自動故障切換、內容雜湊磁碟快取、逐次呼叫的成本/延遲/Token 記帳。一台用 TLA+ 形式化驗證過的安全狀態機在每一次動作前把關; 提示注入偵測在輸入路徑上跑; 一個 EVM 執行層處理鏈上副作用; 一台 RAG 策略 Oracle 負責把 Agent 錨定在我自己的文件庫上。

我一個人做了什麼

端到端。架構、TLA+ 規格、Agent 代碼、Solidity 合約、評測框架, 以及一條命令跑完整個套件的管線。18,400+ 行 Python、7 份 Solidity 合約、14 套測試; 5,866,037 個 TLA+ 狀態, 零違規。

02 · Web3 Chatbot · DeFi

OC2: 鏈上多 Agent 辯論

三個 LLM 角色互相辯論, 然後把判決寫到鏈上。

狀態已交付 · 76 套 Foundry 測試全通過
技術棧Solidity 0.8.23 · Foundry · ECDSA · EVM
架構提議者 · 挑戰者 · 裁判
JD 對應多 Agent · 工具呼叫 · 評測 · Trace

問題

DeFi 動作通常由單一簽名人或小規模多簽把守, 意味著一次失誤的呼叫或一把被攻破的私鑰就能清空一個協議。加一個第二人沒用, 如果他只是給第一個人背書的話。

動機

同行評審比單人評審更強 (獨立的挑戰者強迫提議者為自己的方案辯護) 同一個思路可以讓自主 Agent 比單次 LLM 呼叫更安全。前提是: 你需要密碼學收據, 不是氛圍感, 否則只是演戲。

具體造了什麼

一個三角色辯論模式。Proposer 起草 DeFi 動作, Challenger 攻擊它, Judge 提交判決, 三方都要在辯論紀錄上簽名。完整辯論做雜湊之後寫到鏈上, Foundry 測試通過的 Solidity 棧 (DebateRegistry、JudgeCommitment、SlashingPool、EmergencyStop) 把守執行。任意環節有異常, EmergencyStop 立刻停手, 讓人接手。

我一個人做了什麼

又是端到端。模式、提示工程、Agent 代碼、四份 Solidity 合約、ECDSA 驗證、辯論紀錄雜湊、76 套 Foundry 測試、Gas 基準。最有意思的一步是: 在保留可密碼學驗證審計鏈的前提下, 鏈上成本比 Optimism fault-proof 基線還低 8.7%。

03 · 穩定幣 · IEEE ICBC 2026

MVF-Composer: 穩定幣儲備控制器

一套 12 個 Agent 的穩定幣錨定救援系統, 已被 IEEE ICBC 2026 接收並演講。

狀態已接收 · IEEE ICBC 2026 演講
技術棧Stress Harness · 信任加權 MVF · 3 家 LLM 供應商
規模12 Agent · 1,200 次模擬 · 12,500 行
JD 對應Web3 風控 · 多 Agent 評測 · 對抗魯棒性

問題

穩定幣儲備控制器多在平靜期資料上估協方差。壓力一來, 協方差會被低估 7.17 倍 (“2020 遺漏”), 最優權重反而變成最脆弱的那組, 2020/03 與 2023/03 的脫錨就是這樣發生的。

思路

12 個 LLM Agent 跨 4 類角色 (交易員、做市商、套利者、攻擊者) 在 Stress Harness 內對抗衝擊 (於 t=30 注入)。信任分數 T(a) 對協同或砸盤行為降權, 上層接一個受約束的均值-方差最佳化器, 對壓力增廣後的協方差做再平衡。

具體造了什麼

12 個並發 LLM Agent (5 交易員、3 做市商、2 套利者、2 攻擊者) 跨 OpenAI、Anthropic、DeepSeek 調度, 輸出由 Pydantic 校驗, 進入信任加權風險狀態, 再到受約束的均值-方差最佳化器。1,200 次種子化的黑色星期四模擬跑在消費級硬體上 (每週期 47-99 秒)。

我一個人做了什麼

單作者論文, 獨立工程師系統: Stress Harness、信任加權聚合、壓力增廣協方差混合, 以及 1,200 次可重現模擬。約 12,500 行型別化 Python, 46 個模組。每次執行都記下 seed、commit 與時戳。

04 · 預言機 · 密碼學

NOC: 可密碼學驗證的預言機

鏈上 SNARK 驗證, 成本是現有方案的一小截。

狀態已交付 · 76 項測試 + 256 次模糊
技術棧Solidity 0.8.23 · Groth16 · BN254
拜占庭37.5% 對手下 100% 偵測
JD 對應Web3 基礎設施 · 安全

問題

現有預言機方案要麼信一個小委員會 (便宜, 脆弱), 要麼跑一套重共識 (貴, 慢)。委員會式的設計被攻破過不止一次; 共識式設計把需要逐塊更新的用例價格擠出可用區。

動機

密碼學證明讓你跳過信任假設。驗證器在鏈上跑, 證明本身很便宜, 想撒謊的唯一方式是把曲線攻破。問題是: 能不能在這個想法上搭一個 L2 價格就能跑得起的生產系統。

具體造了什麼

一份 Solidity 0.8.23 預言機, 鏈上跑 Groth16 / BN254 驗證, 上面掛一層質押、罰沒、聲譽加權共識, 用來處理證明拿不到的情況。76 項 Foundry 測試全過, 含 256 次模糊套件和 Gas 基準。37.5% 對手下, 100% 拜占庭偵測率。

我一個人做了什麼

端到端。Solidity 合約、Groth16 驗證器整合、質押與罰沒邏輯、聲譽模型、測試框架、Gas 基準。L2 單次更新成本比現有委員會預言機低 21 倍, 這才是它能用於生產、而不只是寫在論文裡的原因。

Web3 Agent Platform

一個面向 Agent Platform / Web3 Chatbot 方向 JD 的生產級 Agent 平台, 目前在做最後整合。Trace / replay、人在環檢查點、工具註冊表、評測框架已在私有部署上端到端打通, 歡迎申請現場展示。

已經搭好的部分

  • 工具註冊表與策略守門的工具呼叫 (按工具分級的權限範圍)
  • Trace / replay 檢視器, 完整記錄 LLM 呼叫與工具 I/O
  • 動作邊界上的人在環檢查點 (通過 / 改寫 / 拒絕)
  • 評測框架, 含種子化回歸套件, 覆蓋提示與工具變更
  • FastAPI on Cloud Run, Pydantic v2 契約, asyncio 管線
  • LangGraph 編排器, 原生支援 proposer / challenger / judge 角色

為 JD 做的設計

  • 能力分層、模組邊界、權限模型是一等公民
  • Agent 執行: 規劃、工具呼叫、上下文、結果校驗、重試、狀態恢復
  • Harness 級能力: 工具呼叫、工作流引擎、沙箱、評測、Trace, 對 MCP 友善的 I/O
  • SDD 友善: 每次改動都按 spec / test / code 走, 評審按 diff 範圍

為什麼是"進行中"而不是"已發布"

底盤已經搭好。最後 20% 是用真實負載和真實使用者跑一遍, 找出我還沒想到的失敗模式。我寧願寫"進行中", 也不願假裝完工然後靠運氣。
