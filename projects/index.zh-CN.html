OC1: Agent 安全控制系统

多供应商 Agent 栈, 配一个用形式化方法验证过的安全回路。

状态已交付 · 生产级
技术栈Python · FastAPI · TLA+ · Solidity
规模18,400+ 行 · 7 份合约 · 14 套测试
JD 对应Agent Harness · 评测 · Trace · 人在环

问题

前沿模型调用会以三种 Demo 看不到的方式失败: 供应商宕机、提示注入、静默策略漂移。聊天机器人 Demo 不需要处理这些, 但生产级 Agent 系统必须 fail safe, 不能只是 fail loud。

动机

我要的是一个敢把钱和真实用户放上去的底盘。意味着要有一个 可证明 安全的 kill-circuit, 而不是一个希望它安全的回路, 还要让每一次外部调用都可观测、可回滚。

具体造了什么

跨 OpenAI、Anthropic、本地 Ollama 的多供应商 LLM 编排, 带自动故障切换、内容哈希磁盘缓存、逐次调用的成本/延迟/Token 记账。一台用 TLA+ 形式化验证过的安全状态机在每一次动作前把关; 提示注入检测在输入路径上跑; 一个 EVM 执行层处理链上副作用; 一台 RAG 策略 Oracle 负责把 Agent 锚定在我自己的文档库上。

我一个人做了什么

端到端。架构、TLA+ 规格、Agent 代码、Solidity 合约、评测框架, 以及一条命令跑完整个套件的流水线。18,400+ 行 Python、7 份 Solidity 合约、14 套测试; 5,866,037 个 TLA+ 状态, 零违例。

02 · Web3 Chatbot · DeFi

OC2: 链上多 Agent 辩论

三个 LLM 角色互相辩论, 然后把判决写到链上。

状态已交付 · 76 套 Foundry 测试全通过
技术栈Solidity 0.8.23 · Foundry · ECDSA · EVM
架构提议者 · 挑战者 · 裁判
JD 对应多 Agent · 工具调用 · 评测 · Trace

问题

DeFi 动作通常由单一签名人或小规模多签把守, 意味着一次失误的调用或一把被攻破的私钥就能清空一个协议。加一个第二人没用, 如果他只是给第一个人背书的话。

动机

同行评审比单人评审更强 (独立的挑战者强迫提议者为自己的方案辩护) 同样的思路可以让自主 Agent 比单次 LLM 调用更安全。前提是: 你需要密码学收据, 不是氛围感, 否则只是演戏。

具体造了什么

一个三角色辩论模式。Proposer 起草 DeFi 动作, Challenger 攻击它, Judge 提交判决, 三方都要在辩论记录上签名。完整辩论做哈希之后写到链上, Foundry 测试通过的 Solidity 栈 (DebateRegistry、JudgeCommitment、SlashingPool、EmergencyStop) 把守执行。任意环节有异常, EmergencyStop 立刻停手, 让人接手。

我一个人做了什么

又是端到端。模式、提示工程、Agent 代码、四份 Solidity 合约、ECDSA 验证、辩论记录哈希、76 套 Foundry 测试、Gas 基准。最有意思的一步是: 在保留可密码学验证审计链的前提下, 链上成本比 Optimism fault-proof 基线还低 8.7%。

03 · 稳定币 · IEEE ICBC 2026

MVF-Composer: 稳定币储备控制器

一套 12 个 Agent 的稳定币锚定救援系统, 已被 IEEE ICBC 2026 接收并演讲。

状态已接收 · IEEE ICBC 2026 演讲
技术栈12 个 Agent · 4 类角色 · 3 家 LLM 供应商
规模12,500 行 · 46 个模块 · 1,200 次模拟
JD 对应Web3 Chatbot · RAG · 多 Agent 评测

问题

稳定币脱锚是混乱的。USDC 2023/03、UST 2022/05、stETH 2022/06 三场危机的剧情完全不同, 而用某一段历史训出来的控制器往往扛不住下一段。静态仪表盘擅长显示火情, 不擅长灭火。

动机

我想验证一个假设: 一群各看一面的专门 Agent (深度、预言机偏差、跨链价差、情绪) 能不能比单一的整体策略更快、更便宜地协同防御。IEEE ICBC 是这个问题的合适压测场景。

具体造了什么

12 个并发 LLM Agent, 4 类角色 (哨兵、分析师、交易员、治理者), 跨 OpenAI、Anthropic、DeepSeek 调度, 输出由 Pydantic 校验, 上面挂一层受约束的均值-方差优化器。我把真实脱锚事件按区块逐个回放, USDC 2023/03、UST 2022/05、stETH 2022/06, 让这组 Agent 实时反应。

我一个人做了什么

单作者论文, 独立工程师系统。架构、Agent 提示、优化层、评测框架, 以及 1,200 次可复现模拟。12,500 行类型化 Python, 46 个模块。每次运行都记下 seed、commit hash 和时间戳, 任何人都能复现出同样的结果。

04 · 预言机 · 密码学

NOC: 可密码学验证的预言机

链上 SNARK 验证, 成本是现有方案的一小截。

状态已交付 · 76 项测试 + 256 次模糊
技术栈Solidity 0.8.23 · Groth16 · BN254
拜占庭37.5% 对手下 100% 检测
JD 对应Web3 基础设施 · 安全

问题

现有预言机方案要么信一个小委员会 (便宜, 脆弱), 要么跑一套重共识 (贵, 慢)。委员会式的设计被攻破过不止一次; 共识式设计把需要逐块更新的用例价格挤出可用区。

动机

密码学证明让你跳过信任假设。验证器在链上跑, 证明本身很便宜, 想撒谎的唯一方式是把曲线攻破。问题是: 能不能在这个想法上搭一个 L2 价格就能跑得起的生产系统。

具体造了什么

一份 Solidity 0.8.23 预言机, 链上跑 Groth16 / BN254 验证, 上面挂一层质押、罚没、声誉加权共识, 用来处理证明拿不到的情况。76 项 Foundry 测试全过, 含 256 次模糊套件和 Gas 基准。37.5% 对手下, 100% 拜占庭检测率。

我一个人做了什么

端到端。Solidity 合约、Groth16 验证器集成、质押与罚没逻辑、声誉模型、测试框架、Gas 基准。L2 单次更新成本比现有委员会预言机低 21 倍, 这才是它能用在生产、而不只是写在论文里的原因。

Web3 Agent Platform

一个面向 Agent Platform / Web3 Chatbot 方向 JD 的生产级 Agent 平台, 目前在做最后整合。Trace / replay、人在环检查点、工具注册表、评测框架已在私有部署上端到端打通, 欢迎申请现场演示。

已经搭好的部分

  • 工具注册表与策略守门的工具调用 (按工具分级的权限范围)
  • Trace / replay 查看器, 完整记录 LLM 调用与工具 I/O
  • 动作边界上的人在环检查点 (通过 / 改写 / 拒绝)
  • 评测框架, 带种子化回归套件, 覆盖提示与工具变更
  • FastAPI on Cloud Run, Pydantic v2 契约, asyncio 流水线
  • LangGraph 编排器, 原生支持 proposer / challenger / judge 角色

为 JD 做的设计

  • 能力分层、模块边界、权限模型是一等公民
  • Agent 执行: 规划、工具调用、上下文、结果校验、重试、状态恢复
  • Harness 级能力: 工具调用、工作流引擎、沙箱、评测、Trace, 对 MCP 友好的 I/O
  • SDD 友好: 每次改动都按 spec / test / code 走, 评审按 diff 范围
